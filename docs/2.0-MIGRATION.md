# ScreeniesEditor 2.0 — Tauri → Qt Migration Plan

**Goal:** replace the Tauri (WebView2/WebKitGTK) shell with a **Qt** shell so
the app runs on old laptops with no webview, while keeping the **Rust**
backend. The heavy lifting already lives in `core/` (`screenies-core`) —
parser, render/export pipeline, fonts, clipboard — which is deliberately
shell-independent. 2.0 swaps the *shell*, not the engine.

## Architecture decision

**CXX-Qt (KDAB) + QML frontend + `screenies-core` backend.**

| Option | Verdict |
|---|---|
| **CXX-Qt + QML** ✅ | Maintained Rust↔Qt6 bridge. Rust `QObject`s expose `core` to a declarative QML UI. Best fit for a multi-page app (menu / editor / parser / gallery). |
| Qt Widgets via Rust | Painful from Rust; no good binding. Rejected. |
| Slint | Rust-native, no C++/Qt toolchain — but it is **not Qt**. The user asked for Qt specifically. Kept as a fallback if the Qt build chain proves too heavy. |

**Consequences**
- New crate `qt/` (`screenies-qt`) — the 2.0 shell. Depends on `screenies-core`.
- **The Tauri stack (`src-tauri/`, `src/ts/**`, `index.html`, `package.json`,
  `vite.config.ts`, `tsconfig.json`) has been REMOVED** — clean break, no dual
  maintenance. There is therefore **no runnable app until the Qt editor lands
  (phase 2)**. The stable 1.x webview app remains available to users via the
  `main` branch and the `v1.*` release tags; new work happens here on `2.0`.
- **Qt6 is LGPLv3** — we dynamically link and ship the Qt libs + license text.
  Confirm this is acceptable before release (it is for an open, free app).
- Build needs Qt6 + a C++ toolchain (`cmake`, `qmake`) on dev/CI machines.
  Because of that, `qt/` is **excluded from the default cargo workspace**
  (`exclude = ["qt"]`) so `cargo test --workspace` (core + src-tauri) stays
  green on machines without Qt. Build the shell explicitly:
  `cargo run -p screenies-qt` (from inside `qt/`).

## Screens (new information architecture)

```
Main Menu (landing)  ── qt/qml/Main.qml
├─ SSRP Editor       ── the current editor, ported from the TS/canvas UI
├─ Chatlog Parser    ── open a chatlog FOLDER, index + search (NEW)
└─ Gallery           ── browse exported SSRP photos by folder (NEW)
```

## New features (this migration)

1. **Main menu / landing** — hub that routes to Editor, Chatlog Parser, Gallery.
2. **Chatlog Parser + library** — point at a folder of chatlog `.txt` logs; the
   backend indexes every line (reusing `chatlog::parse`) into an in-memory
   store so the user can **search/hover in-app**. Backend: a `chatlog_library`
   module (folder scan + full-text index + query). Persisted index cache so
   re-open is instant.
3. **Effect filters** — `pixelate` and `blur` (and room for more). ✅ **Backend
   done**: `core/src/render/filters.rs` now implements both as neighborhood
   passes, unit-tested, wired into `FilterValues` (`serde(default)` = old
   payloads still parse). Remaining: expose sliders in the Qt UI.
4. **Gallery** — a folder view of previously exported PNGs, with thumbnails and
   "open in editor". Backend: reuse the last-save-dir memory in `config.rs`;
   add a `gallery` module that lists image files + reads thumbnails.

## Backend module map (Rust)

Shell-independent logic lives in `core/` (unit-tested via `cargo test
--workspace` today); the Qt shell only wraps it:

- `core/render/filters.rs` — color ops **+ new `blur`/`pixelate` passes** ✅
- `core/chatlog_library.rs` — folder scan + index + search (feature 2) ✅ logic done
- `core/gallery.rs` — exported-photo folder listing (feature 4) ✅ logic done
- `core/chatlog`, `core/render`, `core/fonts`, `core/clipboard` — unchanged engine
- `qt/src/bridge/app.rs` — CXX-Qt `QObject`(s) that adapt the above to QML.
  Feature `QObject` wrappers (editor/parser/gallery) arrive in phases 2–4.
