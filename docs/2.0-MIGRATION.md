# ScreeniesEditor 2.0 ‚Äî Migration to a native Rust shell (egui)

**Goal:** replace the Tauri (WebView2/WebKitGTK) shell with a **native Rust**
shell so the app runs on old laptops with no webview ‚Äî while keeping every
1.x feature and the Rust engine. The heavy lifting already lives in `core/`
(`screenies-core`): parser, render/export pipeline, fonts, clipboard, and the
new 2.0 additions. 2.0 swaps the *shell*, not the engine.

## Guiding rule: **2.0 is never a downgrade**

Strangler-fig **by branch**: the full-featured **Tauri 1.x app stays on `main`
and the `v1.*` tags** (still downloadable, still the "stable" release) while
`2.0-migration` is the pure native egui future. The TypeScript/Tauri stack has
been removed from *this* branch ‚Äî nothing is lost (it's on `main`), and the two
never have to be maintained side-by-side in one tree. 1.x stays the recommended
download until the egui app reaches parity.

## Architecture decision

**egui / eframe (native Rust) + `screenies-core`.**

| Option | Verdict |
|---|---|
| **egui / eframe** ‚úÖ | Pure Rust, `cargo build` (no C++/Node). Small static binary; runs on old hardware. Already ran `core::compose::render` in the earlier native experiment. Immediate-mode UI is a fine fit for this tool. |
| Slint | Also pure Rust with a software renderer (great compat), declarative. Strong runner-up; picked egui for simplicity + prior proof. |
| Qt / CXX-Qt | Drags in a C++ toolchain + Qt libs (LGPL, heavier). Rejected: contradicts "lightweight + easy build". |
| Tauri (stay) | The webview is the whole problem we're migrating away from. Stays only as the interim shipping app. |

Everything visual is just a skin over `core`, so egui/Slint/Qt would all reuse
the same engine ‚Äî nothing feature-wise depends on the choice.

## Features already in `core` (shell-independent, unit-tested)

- **Effect filters** `blur` + `pixelate` ‚Äî `render/filters.rs`. ‚úÖ
- **Text layout** ‚Äî `render/layout.rs` (wrap + positioning, moved out of the
  old TS canvas; the real measurer `text.rs::GlyphMeasure` sums the same
  ab_glyph advances the renderer pens with ‚Üí preview == PNG by identity). ‚úÖ
- **Chatlog library** ‚Äî folder index + search, `chatlog_library.rs`. ‚úÖ
- **Gallery listing** ‚Äî exported-photo folder scan, `gallery.rs`. ‚úÖ

Because these live in `core`, the native shell reuses them directly.

## Native editor flow (how the preview stays identical to export)

The native shell holds editor state and, on any change:
1. builds `layout::LayoutBlock`s from the chatlog blocks;
2. `text::GlyphMeasure::new(font, size)` ‚Üí `layout::layout_blocks(...)` ‚Üí
   positioned `ExportBlock`s (+ `Bounds` for drag hit-testing);
3. assembles a `RenderJob` and calls `compose::render` ‚Äî the SAME call the PNG
   export uses. The preview is that Rust-rendered image shown in egui, so
   preview and saved file are identical by construction (no separate UI-side
   layout to drift).

## Build / run

- Engine: `cargo test --workspace` (CI ‚Äî `core` only).
- Native shell (excluded from the default workspace): `cd native && cargo run`.
  Needs egui's Linux GUI deps (`libgtk-3-dev`, `libxkbcommon`, `libwayland`,
  GL, ‚Ä¶).
- Preview releases: push a `native-preview-*` tag ‚Üí `native-preview.yml`
  builds `.exe` / `.deb` / `.rpm` and publishes a GitHub pre-release.
